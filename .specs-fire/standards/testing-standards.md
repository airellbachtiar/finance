# Testing Standards

## Overview

Testing effort is concentrated where correctness actually matters financially: the balance engine and currency conversion. UI and CRUD scaffolding get lighter coverage.

## Testing Framework

**Framework**: Vitest
**Runner**: Vitest CLI (fast, works well with TypeScript + Next.js)

## Test Types

| Type | Tool | Location | When to Use |
|------|------|----------|-------------|
| Unit | Vitest | `lib/**/*.test.ts` | Balance engine, currency conversion, split calculation — every non-trivial pure function |
| Integration | Vitest + test DB | `app/api/**/*.test.ts` | Route handlers that touch Prisma (expense creation, settlement recording) |
| Manual/E2E | Browser walkthrough | — | Auth flow, dashboard rendering — verified manually given small user base |

## Coverage Requirements

**Target**: No fixed % target — coverage is judged by whether critical paths are tested, not a number.
**Enforcement**: Not CI-blocking; reviewed at each Validate-mode checkpoint.

**Critical paths that MUST have coverage:**
- Balance engine: equal splits, overridden splits, multi-currency expenses, settlements partially/fully offsetting a balance, mixed EUR/IDR history
- Currency conversion: rounding behavior, missing exchange rate handling
- Household scoping: an Apartment expense must never leak into the Family household balance and vice versa

## Test Naming

**Pattern**: `describe('<unit>') > it('<expected behavior>')`

**Examples**:
- `calculateBalances > splits an equal 3-way expense correctly`
- `calculateBalances > applies a settlement to reduce the payer's balance`
- `convertToEur > throws when no exchange rate is provided for a non-EUR expense`

## Test Structure

```typescript
describe('calculateBalances', () => {
  it('splits an equal 3-way expense correctly', () => {
    const result = calculateBalances([expense], [])
    expect(result).toEqual(expectedBalances)
  })
})
```

## Mock Strategy

**Approach**: Balance engine and conversion logic are pure functions — no mocking needed, just input/output assertions. Integration tests against route handlers use a real (local/test) Postgres instance, not a mocked Prisma client, so schema and query behavior are actually verified.

**Guidelines**:
- Never mock the database for tests that assert financial correctness.

## Test Data

**Strategy**: Hand-written fixtures modeled on the user's real scenarios (€1800 rent 3-way split, €90 groceries, IDR grocery expense with a given exchange rate, a bulk parent settlement).

**Guidelines**:
- Include at least one fixture per household type (Apartment, Family) and one with a non-login member (Mama).

## Running Tests

```bash
# Run all tests
npm run test

# Run with coverage
npm run test -- --coverage

# Run specific test file
npm run test -- lib/balance-engine.test.ts

# Run in watch mode
npm run test -- --watch
```
