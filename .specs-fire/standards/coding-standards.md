# Coding Standards

## Overview

Small, single-developer TypeScript/Next.js codebase. Standards favor readability and low ceremony over enterprise-scale conventions.

## Code Formatting

**Tool**: Prettier
**Config**: `.prettierrc` (2-space indent, single quotes, no semicolons omitted — use default Prettier semicolon behavior)
**Enforcement**: Run on save / pre-commit; not CI-blocking for a solo project.

## Linting

**Tool**: ESLint
**Base Config**: `eslint-config-next` + `@typescript-eslint/recommended`
**Strictness**: Standard — warnings for unused vars/imports, errors for type-unsafe patterns (`no-explicit-any` as a warning, not error, to avoid slowing down solo iteration).

## Naming Conventions

| Element | Convention | Example |
|---------|------------|---------|
| Variables/functions | camelCase | `getHouseholdBalance` |
| React components | PascalCase | `BalanceDashboard` |
| Types/interfaces | PascalCase | `ExpenseSplit` |
| Prisma models | PascalCase singular | `Household`, `Expense`, `Settlement` |
| Database columns | snake_case (via Prisma `@map`) | `created_at` |
| Route handlers | lowercase, REST-ish | `app/api/households/[id]/route.ts` |

### Files and Folders

- **Components**: PascalCase file matching component name (`BalanceDashboard.tsx`)
- **Utilities/lib**: camelCase (`calculateBalances.ts`)
- **Route handlers**: Next.js App Router convention (`app/api/.../route.ts`)

## File Organization

### Project Structure

```
app/                    # Next.js App Router pages + API routes
  api/                  # Route handlers
  (dashboard)/          # Authenticated app pages
lib/                    # Domain logic (balance engine, currency conversion, auth helpers)
prisma/
  schema.prisma         # Data model
  migrations/           # Generated migrations
components/             # Shared React components
```

### Conventions

- **Domain logic lives in `lib/`, not in route handlers** — route handlers stay thin (parse request, call lib function, return response).
- **The balance engine is a pure function** — given a list of expenses and settlements, return computed balances. No side effects, no direct DB writes of balance values.

## Import Order

```typescript
// 1. External packages
import { NextRequest } from 'next/server'
// 2. Internal absolute imports (lib, components)
import { calculateBalances } from '@/lib/balance-engine'
// 3. Relative imports
import { SplitRow } from './SplitRow'
```

**Rules**:
- No circular imports between `lib/` modules
- Prisma client imported from a single shared `lib/db.ts`, never instantiated ad hoc

## Error Handling

### Pattern

**Approach**: Throw typed errors in `lib/` functions; route handlers catch and map to HTTP responses.

### Guidelines

- Never swallow errors silently in financial calculations — a failed conversion or missing exchange rate should surface, not default to 0.
- User-facing errors are plain, actionable messages (e.g. "Exchange rate required for non-EUR expenses"), not raw stack traces.

## Comments and Documentation

### When to Comment

- Only when the WHY isn't obvious from the code — e.g. why balances are always derived and never stored, why a specific rounding rule was chosen for currency conversion.
- No comments restating what a function does when the name already says it.

## Code Patterns

### Preferred Patterns

#### Derived balances, event-sourced style

Balances are never a column that gets updated. They're computed on read from the full history of expenses + settlements for a household. This keeps the audit trail intact and avoids balance drift bugs.

### Anti-Patterns to Avoid

- **Storing a `balance` field on Member/Household and mutating it directly**: breaks the audit trail and is the exact problem this app exists to solve (manual, error-prone balance tracking).
- **Silent currency defaults**: never assume an exchange rate or silently treat IDR amounts as EUR.
